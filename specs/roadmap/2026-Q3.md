## Metodologia de Desenvolvimento Adotada

### 1. Especificação antes do código

Cada feature tem um arquivo `specs/features/FEATURE-XXX.md` com:
- Problem Statement, Proposed Solution
- Acceptance Criteria detalhados
- Detalhamento técnico com contratos de interface (dataclasses, sinais Qt)
- Dependências explícitas entre features

### 2. Agentes especializados por domínio

```
agents/architect.md        → design e contratos
agents/build-engineer.md   → CI/CD, PyInstaller, NSIS
agents/pdf-extractor.md    → extração de metadados
agents/metadata-lookup.md  → busca online
agents/cataloging-engine.md → catalogação CDD
agents/ui-developer.md     → PyQt6, planilha, workers
agents/qa-tester.md        → testes e cobertura
```

Cada agente tem: escopo de arquivos, pré-condições, passo a passo, checklist de entrega.

### 3. Worktrees isoladas para paralelismo

Agentes operam em `git worktree` separadas. Ao mergear:
- Verificar se o agente commitou (alguns não commitaram automaticamente)
- Resolver conflitos em `main_window.py` mantendo adições de ambas as branches

### 4. Testes 100% mockados para código de negócio

- Zero chamadas HTTP nos testes
- Zero acesso a arquivos reais do sistema
- Módulos sem Qt testáveis sem `QApplication`
- Fixtures em `tests/testpath/`

### 5. CI/CD como gate de qualidade

- `git tag vX.Y.Z && git push origin vX.Y.Z` dispara build automático
- Build produz `.exe` portátil + instalador NSIS
- Erros de CI são corrigidos com patch bump (v1.0.0 → v1.0.1 → ...)

## Aprendizados Técnicos do Q3

### PyQt6
- `Qt.ItemFlag` (singular) — `Qt.ItemFlags` não existe em PyQt6
- `BackgroundRole` deve retornar `QBrush`, não `QColor` diretamente
- Workers em `QThread` com `pyqtSignal` para toda I/O bloqueante
- `QProgressDialog` com `setMinimumDuration(0)` para aparecer imediatamente

### CI/CD GitHub Actions
- O runner `windows-latest` usa PowerShell por padrão — `^` não é continuação de linha válida
- NSIS precisa ser instalado explicitamente com `choco install nsis --no-progress -y`
- Pillow é necessário apenas no CI para conversão do ícone (não vai para `requirements.txt`)
- Pinar versões de actions evita warnings de Node.js deprecado (`@v4.2.2`, `@v5.6.0`, `@v2.3.2`)
- `OutFile` no `installer.nsi` deve estar na raiz do projeto (não em subpasta) para bater com o caminho esperado pelo step de Release

### Arquitetura
- Triplicação de código em `main_window.py` foi o principal débito técnico — não estava em `file_manager.py`
- `DualBandTableModel` separado de `FileTableModel` permite evolução sem quebrar código existente
- `SearchPipeline` com 4 estratégias em cascata cobre 95%+ dos casos sem configuração do usuário
- Write-back incremental (`doc.saveIncr()`) do PyMuPDF não recria o PDF — preserva anotações e bookmarks

### Git
- Commits semânticos (`feat:`, `fix:`, `chore:`) facilitam geração automática de changelog
- Tags de patch (`v1.0.1`, `v1.0.2`) são preferíveis a corrigir a mesma tag — mantém histórico de falhas
- Conflitos em `main_window.py` são esperados quando múltiplos agentes adicionam botões em paralelo
